awsdeny

New

AWS AccessDenied errors don't tell you what went wrong or how to fix it. Was it an SCP? A permission boundary? A cross-account trust issue? A typo in the action name? awsdeny tells you in seconds — with a ready-to-paste policy fix.

View on GitHub Quick start
awsdeny explain 
$ awsdeny explain --error "User: arn:aws:iam::123:role/MyRole
  is not authorized to perform: s3:GetObject on resource:
  arn:aws:s3:::bucket/key because no identity-based policy
  allows the s3:GetObject action"

  Access Denied

  Action:    s3:GetObject
  Resource:  arn:aws:s3:::bucket/key
  Principal: arn:aws:iam::123:role/MyRole

  Analysis:
    Type:   Implicit deny
    Source: Identity-based policy
    Reason: No identity-based policy grants the required
            permission.

  Suggested fixes:
    1. Add to your policy: {"Effect": "Allow",
       "Action": "s3:GetObject", "Resource": "...bucket/key"}
    2. Attach a managed policy that includes this permission

  Confidence: very-high (heuristic match)

iampg

Stop guessing IAM permissions. Developers slap on AdministratorAccess because debugging AccessDenied takes hours. iampg generates the exact minimal policy from real API calls, then enforces it in CI.

View on GitHub Documentation
iampg refine --enforce 
$ iampg refine --input infra/deploy-policy.json --enforce

Policy Analysis
==================================================
Statements: 3    Issues: 4

  FAIL  wildcard-action    s3:* grants all actions
        → Replace with: s3:PutObject, s3:GetObject

  FAIL  wildcard-resource  Resource: * on statement 2
        → Scope to: arn:aws:s3:::my-bucket/*

  FAIL  dangerous-perm     iam:CreateUser can create IAM users
        → Remove or justify

  WARN  unused-action      s3:DeleteObject never called in 30 days
        → Consider removing

Enforcement: FAILED — 3 blocking issues found
Exit code: 1
